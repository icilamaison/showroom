# Rule: 작업 목록 실행 및 관리하기

## 목표
`/tasks/`에 저장된 작업 목록 마크다운 파일을 기준으로, 작업을 식별하고 **한 번에 하나씩** 구현하면서 진행 상황을 체크하는 것을 가이드한다.

이 규칙은 3-step 워크플로우의 **Step 3**이다. (Step 1: `create-prd.md`, Step 2: `generate-tasks.md`)

이 단계가 "AI한테 다 알아서 해줘"와 결정적으로 다른 지점이다 — 작업을 몰아서 처리하지 않고, 매 하위 작업마다 멈춰서 사용자 확인을 받는다.

---

## 작업 구현 규칙

### 한 번에 하나의 하위 작업만 진행
- 작업 목록에서 **아직 체크되지 않은 가장 위의 하위 작업 하나**를 찾아 구현한다.
- **다음 하위 작업으로 넘어가기 전에 반드시 사용자의 허락을 구한다.**
- 사용자가 "y", "yes", "ㅇㅇ", "go", "다음" 등으로 명확히 승인하기 전까지는 다음 작업을 시작하지 않는다.
- 한 번에 여러 하위 작업을 동시에 구현하지 않는다. 사용자가 명시적으로 "이번엔 여러 개 한꺼번에 해줘"라고 요청하지 않는 한 1개씩이 기본값이다.

### 완료 처리 프로토콜
하위 작업을 완료할 때마다 다음을 **순서대로** 수행한다:
1. 완료된 하위 작업을 `[x]`로 즉시 표시하고 작업 목록 파일을 갱신한다.
2. 어떤 상위 작업의 **모든** 하위 작업이 `[x]`로 완료되면, 그 상위 작업도 `[x]`로 표시한다.
3. 무엇을 했는지 간단히 보고한다 (변경한 파일, 핵심 로직 요약).
4. 다음 작업으로 진행해도 되는지 사용자에게 묻고 **멈춘다.**

### 작업 목록 파일 유지보수
- 작업을 진행하면서 작업 목록 파일을 그때그때 최신 상태로 갱신한다. (작업이 끝난 뒤 한꺼번에 갱신하지 않는다.)
- 구현 중 PRD에는 없었지만 새롭게 필요해진 작업이 발견되면, 적절한 상위 작업 아래에 새 하위 작업으로 추가한다.
- "Relevant Files" 섹션을 실제로 생성/수정한 파일 기준으로 정확하게 유지한다. 계획에는 있었지만 실제로 안 만든 파일은 제거하고, 계획에 없었지만 새로 만든 파일은 추가한다.

### 막히거나 PRD와 충돌하는 경우
- 구현하다가 PRD나 작업 목록과 실제 상황이 맞지 않는 부분을 발견하면, 임의로 판단해서 진행하지 않고 먼저 사용자에게 알리고 어떻게 할지 확인한다.

---

## AI 에이전트 진행 절차 요약

하위 작업 하나를 처리할 때마다 다음 순서를 따른다:

1. 다음으로 처리할 하위 작업이 무엇인지 명시하고 구현을 시작한다.
2. 하위 작업을 구현한다.
3. 완료 즉시 체크박스를 `[x]`로 갱신하고 파일에 저장한다.
4. 상위 작업의 모든 하위 작업이 끝났는지 확인하고, 끝났다면 상위 작업도 `[x]`로 체크한다.
5. "Relevant Files" 섹션이 최신인지 확인하고 필요하면 갱신한다.
6. 무엇을 했는지 간단히 요약해서 보고한다.
7. **멈추고 사용자에게 다음 작업으로 진행해도 되는지 확인을 받는다.**
8. 사용자가 승인하면 다음 하위 작업으로 돌아가 1번부터 반복한다.

---

## 왜 이렇게 하는가

기능 하나를 통째로 맡기면, 중간에 잘못된 방향으로 가도 끝까지 다 만든 뒤에야 발견된다. 하위 작업 단위로 끊어서 매번 확인받으면, 잘못된 방향을 한 단계 분량의 손실로 막을 수 있다. 이게 이 워크플로우 전체의 핵심 의도다.
