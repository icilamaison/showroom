# Rule: PRD(제품 요구사항 문서) 작성하기

## 목표
AI가 사용자의 짧은 기능 요청을 받아, **주니어 개발자가 그대로 읽고 구현할 수 있는 수준**의 상세한 PRD(Product Requirements Document)를 마크다운으로 작성하도록 가이드한다.

이 규칙은 3-step 워크플로우의 **Step 1**이다.

| Step | 이름 | 규칙 파일 |
|---|---|---|
| 1 | Create PRD | `create-prd.md` (현재 파일) |
| 2 | Generate Tasks | `generate-tasks.md` |
| 3 | Execute Tasks One by One | `process-task-list.md` |

---

## 진행 과정

1. **기능 요청 받기**: 사용자가 새 기능에 대해 짧게 설명한다. (한두 문단 수준이어도 충분하다.)
2. **명확화 질문하기 (PRD 작성 전, 필수)**: 바로 PRD를 작성하지 않는다. 먼저 누락된 정보를 확인하기 위한 질문을 한다.
3. **답변 반영 후 PRD 작성**: 사용자의 답변을 받은 뒤에만 PRD 본문을 작성한다.
4. **파일로 저장**: 결과물은 `/tasks/prd-[기능명].md` 형식으로 저장한다. (예: `/tasks/prd-contract-system.md`)

> 중요: 1~2번을 건너뛰고 곧바로 완성된 PRD를 내놓지 않는다. 질문 없이 바로 작성된 PRD는 가정이 너무 많이 들어가 실제 요구사항과 어긋날 위험이 크다.

---

## 명확화 질문 가이드

질문은 **3~5개**로 제한한다. 답변하기 쉬운 형태(선택형, 단답형)로 묻는다. 다음 카테고리를 참고해 상황에 맞는 질문을 고른다 — 전부 다 물을 필요는 없다.

- **문제/목표**: 이 기능이 해결하려는 핵심 문제는 무엇인가?
- **타겟 사용자**: 누가 이 기능을 사용하는가? (고객/관리자/내부 직원 등)
- **핵심 동작**: 사용자가 어떤 행동을 할 수 있어야 하는가?
- **범위**: 이번 1차 개발에 포함할 것과 명시적으로 제외할 것은?
- **데이터**: 어떤 데이터를 저장/조회해야 하는가? 기존 데이터와 연결되는가?
- **인증/권한**: 로그인이 필요한 기능인가? 누구나 접근 가능한가?
- **디자인/UI**: 기존 디자인 시스템이나 컴포넌트를 따라야 하는가?
- **엣지 케이스**: 예외 상황, 실패 시나리오, 동시성 이슈 등을 어떻게 처리할 것인가?

---

## PRD 구조

생성하는 PRD는 다음 섹션을 포함해야 한다. 기능 성격에 따라 일부 섹션(디자인 고려사항 등)은 생략할 수 있다.

1. **개요** — 무엇을 만드는지 1~2문단으로 요약
2. **배경 및 문제 정의** — 왜 필요한지
3. **목표** — 이 기능이 달성해야 하는 것 (측정 가능하면 좋음)
4. **User Story** — "~로서, 나는 ~하고 싶다. 그래야 ~할 수 있다" 형식
5. **기능 요구사항 (Functional Requirements)** — 번호를 매겨 "시스템은 반드시 ~해야 한다" 형식으로 명확하게
6. **비기능 요구사항 (Non-Functional Requirements)** — 보안, 성능, 데이터 정합성 등
7. **범위** — 포함(In Scope) / 제외(Out of Scope)를 표로 명확히 구분
8. **데이터 모델** — 필요 시 테이블 스키마, 컬럼 설명
9. **API 명세** — 필요 시 엔드포인트, request/response 예시
10. **Acceptance Criteria** — 체크리스트 형태, "이 조건을 만족하면 완료"
11. **향후 확장 가능성** — 이번엔 제외했지만 나중에 붙일 것들과, 그걸 고려한 구조적 여지
12. **Open Questions** — 아직 결정되지 않아 추가 확인이 필요한 사항

---

## 대상 독자 및 작성 원칙

- PRD는 **주니어 개발자**가 추가 설명 없이 읽고 바로 작업으로 옮길 수 있어야 한다.
- "적절히", "필요시", "유연하게" 같은 모호한 표현을 피하고, 구체적인 규칙·숫자·예시를 쓴다.
- 기능 요구사항과 비기능 요구사항을 분리해서 작성한다.
- 제외 범위는 빠뜨리지 말고 반드시 명시한다. (포함 범위만 적으면 "이것도 해주는 거 아니었어?" 하는 스코프 크립이 생긴다.)

---

## 최종 지침

1. 명확화 질문에 대한 답변을 받기 전까지 PRD 작성을 시작하지 않는다.
2. PRD 작성 후 곧바로 코드 구현으로 넘어가지 않는다. PRD가 완성되면 Step 2(`generate-tasks.md`)로 넘어간다.
3. 완성된 PRD는 `/tasks/prd-[기능명].md`로 저장한다.
